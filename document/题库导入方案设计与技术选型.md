# 题库导入方案设计与技术选型

> **数据模型以《数据模型总表》为准**；本文档聚焦题库导入的设计与技术选型。

> 本文档为刷题系统 PRD 的组成部分,聚焦"如何把本地大量题库导入数据库"这一核心功能。涵盖整体设计思路、格式技术选型、性能约束、Excel 导入模板设计、字段定义与数据模型、导入处理管线。

---

## 1. 背景与范围

刷题系统需要把教师手中大量、来源分散、格式粗糙的题库,稳定高效地导入到 MySQL 数据库中,供后续刷题、组卷、统计使用。

**技术栈**:Next.js(全栈)+ Prisma + MySQL。

**数据规模**:单次导入题量最大量约 500–1000 题(对本方案而言属轻量级)。

**题型范围(本期)**:全部为**纯文字题型**,共四种——单选题、多选题、判断题、多空填空题。简答题因涉及人工批改,移至二期。不含图片、公式、音视频及组合/理解等富媒体题型(此类题型若未来需要,另走 Word/富媒体通道,不在本期范围)。

---

## 2. 整体设计思路:格式双层分离

本方案的核心决策是:**不要在"Word / Excel / CSV"中三选一作为唯一格式,而是把"人工填写的模板"与"系统真正吃进去的格式"拆成两层。**

- **人工授权层(给人用)**:Excel(`.xlsx`)模板。教师在此填写/校对题目,享受多 sheet、下拉校验、批注提示等"脚手架"能力。
- **系统规范层(给机器用)**:JSON(canonical format)。这是系统内部唯一认可的结构化格式,所有校验、入库逻辑只面向它。

两者之间由一个**解析器**衔接,整体数据流为:

```
Excel 文件 ──解析──▶ 归一化 JSON ──Zod 校验──▶ createMany ──▶ MySQL
                          ▲
            (AI 转换流水线也可直接产出此 JSON)
```

这样设计的收益:

1. **单一数据契约**。JSON schema 同时是"AI 输出的契约"和"模板解析的目标",人和 AI 两条入口最终汇聚到同一套校验与入库逻辑,数据模型只维护一份。
2. **模板与数据库解耦**。模板列不必和数据库字段 1:1。模板可以保持精简,数据库可以富余预留,二者独立演进。
3. **AI 转换天然契合**。把杂乱原始题库交给 AI(借助文档处理能力)直接产出符合规范的 JSON 或标准 Excel,跳过人工重排版这一最痛环节。

---

## 3. 技术选型:为什么是 Excel + JSON

主流在线考试系统(优考试、轻速云、云帆、磨题帮等)普遍同时提供 Excel / Word / Txt 模板。其中关键差异是:Word 模板支持图片、音视频、公式及组合/理解题,而 Excel 模板不支持这些富媒体内容,但对纯文本规整客观题的解析最为稳定可靠。由于本期题型全部为纯文字,Excel 正是最契合的选择。

四种候选格式横向对比:

| 维度 | CSV | Excel (.xlsx) | Word | JSON |
|---|---|---|---|---|
| 中文 / 逗号 / 换行 | 易踩编码与引号坑 | 原生无忧 | 无忧 | 无忧 |
| 多题型分表 | 不支持 | 多 sheet 天然分型 | 靠标记符 | 结构化字段 |
| 下拉校验 / 填写提示 | 无 | 数据有效性 + 批注 | 弱 | 无(面向机器) |
| 公式 / 截图 / 音视频 | 无 | 无 | 支持 | 需转 base64/URL |
| 解析难度(给程序) | 中 | 低(列固定) | 高(需 NLP/AI) | 极低 |
| 教师上手度 | 低 | 高 | 最高 | 不面向人 |

**CSV 不作为主模板**:在中文题库场景里它"看似简单实则最坑"——题干内的逗号或换行会撕裂列,Windows 下 Excel 打开 UTF-8 存在 BOM 问题,且无法多 sheet 分题型、无法做下拉校验。CSV 仅作为导出兜底。

**Excel 作为人工模板的真正价值**不在文件格式本身,而在于它能承载"给人用的脚手架":题型/难度列做成下拉(数据有效性)、首行冻结、批注写明填写规则、错误行高亮。这与商业系统强调的"输入区 / 检查区"双栏预检体验一致——导入前先智能检测可识别题数、对异常题给出提示、并做查重。

**JSON 作为系统规范格式**:解析后归一、AI 直接产出,两条入口共用同一套 Zod 校验,数据模型只需一份。

## 4. 性能与技术约束(Prisma + MySQL + Next.js)

500–1000 题对 MySQL + Prisma 属毫秒级写入,**真正影响体验的是解析与校验,而非写库**。但需规避以下典型反模式:

1. **必须用 `createMany`,禁止循环 `create`**。`createMany` 生成单条批量 INSERT,显著减少数据库往返;循环逐条写是连接池杀手,会迅速耗尽连接导致超时与诡异错误。
2. **分批,但本量级一批即可**。Prisma 官方建议大批量写入分批(如每批 1000 条)。本期单次 500–1000 题,一个 `createMany` 即可;保险起见可按 500/批切两批。
3. **去重用 `skipDuplicates`**。`createMany({ skipDuplicates: true })` 可跳过违反唯一约束的记录,且**在 MySQL 上受支持**。配合内容哈希(见下)实现幂等导入。
4. **避免分类/知识点的 N+1**。不要逐行 `findOrCreate` 分类。正确顺序:先去重收集本批涉及的分类/标签 → 一次性 `createMany(skipDuplicates)` 建好 → 查回 id 映射 → 再连同 id 一起 `createMany` 题目。
5. **PrismaClient 单例**。多实例会耗尽连接池。Next.js 中务必挂 `globalThis` 复用,否则 dev 热重载会爆连接。
6. **部署侧**:本项目部署为国内云服务器上的长驻 Node 进程(Docker),不存在 Serverless 执行时长/请求体限制；此条仅作通用提示。1000 题的 xlsx 文件解析 + 校验 + 入库可在单个 Route Handler / Server Action 内同步完成;待规模上升到数千上万题再引入后台任务队列。

**幂等去重设计**:为每道题计算 `contentHash`(题型 + 归一化题干 + 答案 做 SHA1),设为唯一索引。重复导入时由 `skipDuplicates` 自动跳过,并将跳过题数回显给用户。

## 5. 导入模板设计

### 5.1 单 Sheet 方案

四种题型均为纯文本、字段高度重合,故采用**单个 sheet + 一列"题型"下拉**,而非四个 sheet。理由:解析只需一个循环,教师只看一张表,摩擦最小。(若有"判断题不应出现选项列"的洁癖,可改为多 sheet 分型,解析逻辑一致。)

### 5.2 列结构

```
题型 | 题干 | 选项A | 选项B | 选项C | 选项D | 选项E | 选项F | 答案 | 难度 | 解析 | 知识点 | 教材单元 | 教材 | 外部编号
```

模板增强项:题型列与难度列用**数据有效性下拉**;首行冻结并加**批注**写明规则;难度列**预填默认值"中"**;选项列预留 A–F 六个(够用且不冗余)。

### 5.3 答案列编码约定(关键)

四种题型（一期）共用一列"答案",靠以下约定区分,务必写进模板批注：

| 题型 | 答案列填写方式 | 入库后 JSON（answer） | options 格式 |
|---|---|---|---|
| 单选 | `B` | `["B"]` | `[{key:"A",text:"…"},…]` |
| 多选 | `ABC` 或 `A,B,C` | `["A","C"]`（以实际选项为准） | `[{key:"A",text:"…"},…]` |
| 判断 | `对` / `错`（亦可 √ / ×） | `["T"]` / `["F"]` | `null` |
| 多空填空 | 各空用 `###` 分隔；单空内多个可接受答案用 `\|\|` 分隔 | `[["北京","京城"],["上海"]]` | `null` |

> 简答题因涉及人工批改，移至二期，一期不实现。

answer 一律为**数组**（含单选），与《数据模型总表》§6 保持一致。

**填空示例**:三个空,第一空"北京/京城"均算对 → 答案列填
`北京||京城###上海###广州`

### 5.4 题干占位符约定

填空题的题干中,**用统一占位符标空**(如连续下划线 `____` 或 `{{1}}`)。解析时通过数占位符个数,校验其与答案空数(`###` 段数)是否一致——可在预检阶段拦截大量"答案数对不上"的脏数据。

## 6. 字段定义

### 6.1 必填规则分层

必填规则**不是全局统一的,而是分两层**:

- **全局必填(任何题型)**:`题型`、`题干`。仅此两项。
- **条件必填(随题型而定)**:
  - `选项 A/B`(至少两个)→ 仅单选 / 多选;判断、填空留空。
  - `答案` → 一期四种题型（单选 / 多选 / 判断 / 填空）均必填；填空按占位符数校验答案段数。

因此"选项"不能简单设为全局必填,否则判断题永远过不了校验;校验器须按 `switch(题型)` 分流处理。

### 6.2 字段清单与归属

| 列 / 字段 | 模板 | 必填性 | 说明 |
|---|---|---|---|
| 题型 | ✓ | **全局必填** | 下拉:单选 / 多选 / 判断 / 填空（一期四种；简答移至二期） |
| 题干 | ✓ | **全局必填** | 填空题用占位符标空(如 `____`) |
| 选项 A–F | ✓ | 条件必填 | 仅单选 / 多选;其余留空 |
| 答案 | ✓ | **条件必填（一期四种均必填）** | 按 5.3 编码；answer 一律数组格式 |
| 难度 | ✓ | 选填(默认 MEDIUM) | 下拉：EASY / MEDIUM / HARD；枚举值，空则填默认 MEDIUM |
| 解析 | ✓ | 选填 | |
| 知识点 | ✓ | 选填 | 三级归属之一 |
| 教材单元 | ✓ | 选填 | 三级归属之一 |
| 教材 | ✓ | 选填 | 三级归属之一 |
| 外部编号 | ✓ | 选填 | 原始资料编号,利于去重与"重导更新" |
| 标签 tags | ✗(仅 schema) | — | 横切自由标签(如"易错题""期末重点") |
| 分值 score | ✗(仅 schema) | — | 刷题不用,进试卷时需要,预留 |
| 导入批次 importBatchId | ✗(仅 schema) | — | 后端字段,支持整批回滚/删除 |
| contentHash | ✗(仅 schema) | — | 去重唯一索引,系统自动计算 |

### 6.3 "为后续留空间"的正确姿势

留空间应**留在 schema 里,而非堆在模板里**:

- **模板只放"人现在真会填"的列**,保持精简(空列过多会吓退教师、降低使用率)。
- **schema 多预留 nullable 字段**(空字段几乎零成本)。
- **解析器对模板中缺失的列一律按 null 处理**。

如此,将来某功能真的需要某字段(如分值)时,只需在模板加一列——schema 字段早已存在,**老数据、老模板无需改动**即可继续导入。

### 6.4 关于"难度"给默认值

难度若允许空,后续"按难度筛题 / 组卷 / 难度分布统计"会遍布 `null` 需处处兜底。做法:模板用下拉并**预填默认 MEDIUM**——名义上选填(不改也能导),但永远非空。难度枚举为 `EASY / MEDIUM / HARD`（对应数据库 `Difficulty` 枚举），与《数据模型总表》§0 一致。

### 6.5 教材 → 单元 → 知识点 的三级归属

三者构成一条层次链(教材 → 单元 → 知识点)。当前阶段:**模板平铺为三列**(填写最简),**数据库先用三个 nullable 字符串字段冗余存储**。待将来需要"按教材/单元树形浏览题库"时,再规范成分类树(category tree)。一上来就建分类树属过度设计。

## 7. 数据模型(Prisma 草案)

选项与答案因题型结构差异大,统一用 MySQL **JSON 列**(Prisma `Json` 类型)存储,避免硬拆关系表的别扭。

```prisma
model Question {
  id            Int          @id @default(autoincrement())
  bankId        Int
  bank          QuestionBank @relation(fields: [bankId], references: [id], onDelete: Cascade)
  // 题目必须归属题库；bankId 来自当前题库页面上下文（与学生导入的 classId 对称），不在 Excel 模板中

  type          QuestionType               // SINGLE_CHOICE | MULTIPLE_CHOICE | TRUE_FALSE | FILL_BLANK（一期四种；SHORT_ANSWER 简答为二期）
  stem          String       @db.Text      // 题干；填空用占位符标空（如 ____ 或 {{1}}）
  options       Json?                      // [{key:"A",text:"..."}]；判断/填空为 null
  answer        Json                       // 一律数组化，见 §5.3 答案编码（单选["B"]、多选["A","C"]、判断["T"]/["F"]、填空[["北京","京城"],["上海"]]）
  difficulty    Difficulty   @default(MEDIUM)
  analysis      String?      @db.Text
  score         Int?         // 预留,刷题阶段不用;权威分值在 ExamQuestion

  // 三级归属(先平铺冗余,后续可规范为分类树)
  textbook      String?
  unit          String?
  knowledgePoint String?

  tags          String?      // 预留,简单实现可先存逗号分隔串,后续转多对多
  externalId    String?      // 外部编号，支持"重导更新"
  contentHash   String       // 去重：题型+归一化题干+答案 的 SHA1
  importBatchId String?      // 导入批次,支持整批回滚

  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @updatedAt

  @@unique([bankId, contentHash])          // 库内去重；允许同题进多个题库（原 contentHash @unique 为全局唯一，改为库内唯一以支持同题入多库场景）
  @@index([type])
  @@index([difficulty])
  @@index([importBatchId])
}
```

> 备注：`tags` 初期可用逗号分隔字符串快速落地,需要按标签精确检索/统计时再升级为独立标签表 + 多对多关联。去重键由原先的 `contentHash @unique`（全局唯一）改为 `@@unique([bankId, contentHash])`（库内去重），原因：允许同一道题被导入到多个不同题库，符合实际使用场景（如公共题库与专项题库共享同题）。

## 8. 导入处理管线

四步流程,**必须包含 dry-run 预检**(对标商业系统的"检查区"):

1. **上传 → 解析**。Route Handler / Server Action 接收 `.xlsx`,用 SheetJS(`xlsx` 库)解析为行数组;AI 通道则直接接收 JSON。**`bankId` 来自当前题库页面上下文（与学生导入的 `classId` 对称），不在 Excel 模板中**——由服务端从 URL 参数或 Session 读取，注入每行数据。
2. **逐行 Zod 校验(不写库)**。按题型分流校验,收集每行错误并附行号;向前端返回"可导入 N 题 / 异常 M 题"预览,异常项给出可读提示供修正。
3. **确认导入**。先去重建好分类/标签,再 `createMany({ data, skipDuplicates: true })`;需原子性时包 `$transaction`,并打上本次 `importBatchId`。
4. **查重反馈**。基于 `[bankId, contentHash]` 跳过的题数回显给用户。

### 8.1 按题型的校验规则

| 题型 | 校验要点 |
|---|---|
| 单选 | 答案为单个有效选项字母（解析后入库为 `["B"]`）；至少 2 个选项 |
| 多选 | 答案为 ≥2 个有效选项字母（入库为 `["A","C"]` 等）；至少 2 个选项 |
| 判断 | 答案仅限约定取值（对/错、√/×），解析后入库为 `["T"]` / `["F"]`；无选项 |
| 填空 | 题干占位符个数 == 答案 `###` 段数；每空答案非空 |

通用规则:`题型`、`题干` 非空;`题型` 取值在枚举内（SINGLE_CHOICE / MULTIPLE_CHOICE / TRUE_FALSE / FILL_BLANK）;`难度` 取值在枚举内（EASY / MEDIUM / HARD，空则填默认 MEDIUM）。

## 9. AI 辅助:把粗糙题库转为模板

针对"大量题库来源杂乱(Word/PDF/网页文本)、人工重排版最痛"的现实,提供两条转换流水线:

- **流水线 A(直出 JSON,推荐技术向使用)**:读取杂乱原文 → 按 JSON schema 直接产出结构化题目 → 走 JSON 导入口,跳过 Excel 落地。
- **流水线 B(产出标准 Excel,供非技术教师协作)**:读取原文 → 输出符合本模板规范的 `.xlsx`(多 sheet/下拉/批注)→ 教师在 Excel 二次核对 → 走 Excel 导入口。

设计精妙处:**模板规范同时是"AI 输出契约"与"人工填写规范"**,人与 AI 共用同一份 schema,系统只认它。

---

## 10. 后续扩展预留

- **富媒体题型**:图片/公式/组合题另开 Word 或富媒体通道,题目内图片走对象存储 + URL 引用,不影响本期纯文本主链路。
- **重导更新**:借助 `externalId`,将来可支持"按外部编号匹配已有题做 update"而非新增。
- **分类树**:`textbook/unit/knowledgePoint` 三字段后续可规范为 category tree,实现树形浏览。
- **标签体系**:`tags` 由逗号串升级为多对多关联表。
- **批量运维**:基于 `importBatchId` 实现整批回滚、批量改难度等。
