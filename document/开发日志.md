## [2026-06-30] 修复 — 班级共享模型重构 ✅

### 背景（用户反馈的 S2 缺陷）
1. 可创建同名班级（组卷选班会混淆）。2. 班级被创建者独占，其他老师看不到/无法选用。3. 担心删班级级联删学生、删后无法重新导入。

### 数据库诊断
- 根因：`Class.teacherId` 把班级建成"被某老师独占的资源"，而班级本应是**全校实体**。问题 1、2 是同一根因的两个症状（独占 → 名字不全局唯一 + 可见性按创建者过滤）。`学生导入方案设计` 原意本就是"班级共享、其他老师复用"，是 S2 实现加了按老师过滤而背离。
- 问题 3 **不是 bug**：`Enrollment` 的 `onDelete: Cascade` 在成员关系上，删班只清 Enrollment，**学生身份保留**；删后学生变游离学生，重新建班/加授课后**可被重新导入**（姓名一致→importable；不一致→姓名冲突提示，非永久阻塞）。

### 方案（已与用户拍板）
全校共享模型 + 授课 M2M：
- `Class.name @unique`；`Class.teacherId` 保留，语义改为**创建者**（仅审计 + 改名/删班鉴权）
- 新增 **`ClassTeacher`** 授课关联表（与 Enrollment 平行）；"我授课的班级"=该表口径
- 班级管理新增**「班级列表」**弹窗（看全校班级 + 「添加授课」把已存在班级加入我的授课）；新建班级自动把创建者设为授课

### 实现
- schema + migration `20260630120000_class_teaching_sharing`（diff 生成 SQL 后 `migrate deploy`，规避非交互限制）；seed 增 ClassTeacher（教师1授课两班、教师2共享高一1班）

## [2026-07-01] Bugfix — 学生端「查看成绩」点击无效

### 问题现象
学生端练习/考试列表中，已完成且不可重做的卡片（`status === "SUBMITTED" && !canRepeat`）按钮显示「查看成绩」，但点击无反应，无法回到已实现的结果页（`/student/exams/[id]/result`）查看成绩和错题展示。

### 根因
`src/components/student/exam-list.tsx` 的 `ExamCard` 中：
1. `buttonDisabled = exam.status === "SUBMITTED" && !exam.canRepeat` —— 这恰好就是「查看成绩」出现的状态，等于把这颗按钮直接禁用了，点击自然无效。
2. 即便不禁用，`onClick` 统一绑定的是 `onStart`（父组件 `handleStart` 只会 `router.push('/student/exams/{id}')`，即重新进入做题页），从未指向已实现的结果页路由。
3. 列表接口 `GET /api/student/exams` 只返回 `latestScore`，没有把最近一次已提交 `Attempt.id` 传给前端，前端即使想跳转结果页也拼不出 `?attemptId=` 参数。

### 修复
- `src/app/api/student/exams/route.ts`：`latestSubmitted` 基础上新增字段 `latestAttemptId: latestSubmitted?.id ?? null`，随列表项一起返回。
- `src/components/student/exam-list.tsx`：
  - `ExamItem` 接口新增 `latestAttemptId: number | null`
  - `handleStart` 改名为 `handleAction`：当 `status === "SUBMITTED" && !canRepeat` 时 `router.push('/student/exams/{id}/result?attemptId={latestAttemptId}')`（即已有的结果页），否则维持原逻辑跳做题页（新建/续答/重做）
  - 移除 `buttonDisabled`（该状态本身有明确可执行动作——查看成绩，不应被禁用），按钮恢复始终可点击的蓝色实心样式
- 未新建任何 UI，结果页 UI（分数环、逐题对错展开）在 S8 已完整实现，仅补上入口链接。

### 验证
- `npx tsc --noEmit` 通过
- 全仓 grep `查看成绩` 确认只有这一处按钮文案，无其它遗漏入口

### 本次决策
- 「已完成且不可重做」不再等同于「禁用按钮」，而是一个有效动作（查看成绩），按钮可用性应由「有无可执行动作」决定，而非单纯由 `canRepeat` 推断。

### 遇到的问题及解决方案
- 问题：单镜像内 `prisma migrate deploy` 报 `Cannot find module 'effect'`。Prisma 6 CLI 的 `@prisma/config` 依赖被提升到顶层的传递依赖（`effect` 等），只 COPY `node_modules/prisma`+`@prisma`+`.prisma` 不够。
  解决：**迁移与运行分离**——迁移交给独立 `migrate` 服务（`migrator` 阶段，完整依赖，CLI 可用），跑完退出；`app` 镜像因此免带 CLI，更精简（≈560MB）。
- 问题：Prisma 在 alpine 上 musl/openssl 引擎易出兼容问题。
  解决：全程 `node:22-slim`（Debian glibc + openssl3），零折腾；runner 额外 `COPY .prisma` 补齐 standalone 偶漏的查询引擎。
- 问题：本机 80 端口被其他容器占用，nginx 起不来；且 compose 对 `ports` 是**追加合并**，普通 override 无法替换。
  解决：验证时用 `-f docker-compose.yml -f override.yml`，override 里 `ports: !override [ "8080:80" ]` 强制替换，在 8080 跑通全链路。
