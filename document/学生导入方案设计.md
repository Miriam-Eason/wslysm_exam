# 学生导入方案设计

> **数据模型以《数据模型总表》为准**；本文档聚焦学生导入的设计思路与流程，schema 片段已同步至总表口径。

> 本文档为刷题系统 PRD 的组成部分,聚焦"教师创建班级、并向班级批量导入学生"这一功能。涵盖整体设计思路、数据模型、学号字段设计、导入处理逻辑、冲突与边界处理、批量删除、账号与默认密码,以及面向后续"竞赛班/跨班"功能的冗余预留。

---

## 1. 背景与目标

教师端可创建班级,并在班级内通过 Excel 模板批量导入学生。导入文件只含**学号**与**姓名**两项信息;学生归属的班级由"当前所在班级页面"的上下文决定,不写入 Excel。

**一期目标(本期实现)**:

- 学生只属于唯一一个行政班(一个学生 = 一个班)。
- 同一班级由**第一位老师**完成学生导入即可,其他老师直接复用已建好的班级,无需重复导入。

**后续迭代(本期不实现,但需埋好设计)**:

- 可能出现竞赛班等场景,**个别学生同时存在于多个班级**。
- 要求:扩展该功能时,**数据库结构与导入功能不需要大改**。

---

## 2. 核心设计思路:多对多建模 + 应用层约束

本方案的关键决策是:**数据库底层直接采用"学生 ↔ 班级"多对多模型,而一期"一个学生只能在一个班"的限制,由应用层(导入校验)实现,不写死在数据库结构里。**

冲突问题的根因在于:容易把"学生身份"与"班级成员关系"当成同一件事。一旦把二者拆开,冲突自然消失:

- **学号是一个全局唯一的"人"(身份)**。
- **"在哪个班"是这个人与班级之间的一段关系(成员关系)**。

由此推导出三张表,而非两张:`Student`(身份)、`Class`(班级)、`Enrollment`(成员关系中间表)。

**为什么一期不把"唯一班级"写死进数据库?** 若一期在 `Student` 上挂 `classId`、或对 enrollment 施加"一个学生只能一条"的约束,后续做竞赛班时就必须改表结构、迁移数据、重写导入逻辑——正是要避免的"大改"。多对多结构天生支持多班,一期只是**自愿不去用这个能力**;后续放开,等价于"去掉一道应用层校验",而非改数据库。

---

## 3. 学号字段设计

学号为 **9–10 位数字组成的唯一值**,但**务必以字符串(String / VARCHAR)存储,而非数值类型(Int/BigInt)**。原因:

- 学号是**标识符**,不参与任何数值计算。
- 以 0 开头的学号用数字类型会**丢失前导零**。
- 10 位数字已逼近 Int 上限,语义上也不应当作数字处理。

学号唯一约束作用于**身份层(Student.studentNo)**,与班级无关——这是"同一学号进多个班也不冲突"的前提。

---

## 4. 数据模型(Prisma 草案)

```prisma
model Student {
  id           Int          @id @default(autoincrement())
  studentNo    String       @unique @db.VarChar(10)   // 9-10位学号,全局唯一身份
  name         String
  passwordHash String                                 // 存默认密码的哈希,绝不存明文
  mustResetPwd Boolean      @default(true)            // 冗余预留:是否仍在用默认密码 / 是否需强制改密
  deletedAt    DateTime?                              // 冗余预留:软删除标记(一期不强制使用)
  enrollments  Enrollment[]
}

model Class {
  id          Int          @id @default(autoincrement())
  name        String
  teacherId   Int          // 哪个老师创建的
  createdAt   DateTime     @default(now())
  enrollments Enrollment[]
}

model Enrollment {          // 中间表:谁在哪个班
  id        Int      @id @default(autoincrement())
  studentId Int
  classId   Int
  // role   String   @default("regular")  // ← 冗余预留:后续竞赛班用 "regular"/"contest" 区分关系类型
  student   Student  @relation(fields: [studentId], references: [id], onDelete: Cascade)
  class     Class    @relation(fields: [classId], references: [id], onDelete: Cascade)

  @@unique([studentId, classId])   // 同一人不在同一班重复登记(一期与后续均成立)
  @@index([classId])
}
```

约束职责划分:

- **`Student.studentNo @unique`**:保证同一学号在系统内只有一条学生记录(身份唯一)。**该约束不与班级绑定**,故学号进多少个班都不冲突。
- **`@@unique([studentId, classId])`**:保证同一个人在同一个班里只登记一次。该规则在任何阶段都成立,一期与后续都无需改动。
- **`Enrollment` 两个关系均配 `onDelete: Cascade`**:删除 Student 或 Class 时,其关联的 Enrollment 由外键级联自动清除,避免孤儿关系记录(详见第 8 节"批量删除")。
- **注意**:模型中**没有任何"一个学生只能属于一个班"的数据库约束**——该规则一期由导入校验(应用层)实现(见第 5 节)。
- **`Enrollment.role`(注释预留)**:后续竞赛班只需启用此字段,用 `regular`/`contest` 区分关系类型,字段位置已预先占好。
- **`passwordHash` / `mustResetPwd` / `deletedAt`**:账号与软删除相关字段,见第 8、9 节。

## 5. 导入处理逻辑

教师在某个班级页面内执行导入,因此 **classId 来自页面上下文,不在 Excel 中**;Excel 模板仅含学号、姓名两列。

处理流程采用批量(bulk)模式,避免逐行 N+1:

1. **解析 + 文件内去重**。读出文件全部学号,在内存中先行去重;若老师自己填重了,提示重复的行号。
2. **查已存在数据**。`findMany({ where: { studentNo: { in: [...] } } })` 查出已存在的学生,并查出他们已有的 `Enrollment`(已属哪个班)。
3. **一期专属校验:已有班级则跳过(A 方案)**。某学号若已属于其它班级 → **本次不导入该生**,在 dry-run 预检中提示"学号 XXX 已属于【班级 A】,本次跳过"。
4. **新建学生身份**。文件中尚不存在的学生 → `createMany({ skipDuplicates: true })` 批量创建。
5. **建立成员关系**。批量 `createMany({ data: enrollments, skipDuplicates: true })`,由 `[studentId, classId]` 唯一约束兜底去重。

该流程的良好性质:

- **第一位老师导完,其他老师直接复用**:其他老师进入的是同一个班级(同一 classId),本就不会触发任何导入冲突,这是模型的自然结果。
- **幂等**:同一位老师对同一班级重复导入同一份文件,关系已存在即被跳过,不产生脏数据。

## 6. 冲突与边界处理

| 场景 | 处理方式 |
|---|---|
| 学号已属于别的班级 | **一期采用 A 方案:直接跳过 + 预检提示**(导入继续,这几人不动)。转班等低频操作由老师手动处理 |
| 学号已存在但姓名不一致(如 001 原为"张三",文件写"张三丰") | **不静默覆盖**。dry-run 预检中提示"学号 001 已存在,姓名为张三,文件中为张三丰",由老师确认是否更新姓名 |
| 同一班级重复导入同一文件 | 靠 `[studentId, classId]` 唯一 + `skipDuplicates`,天然幂等 |
| 文件内学号重复(老师自己填重) | 解析阶段内存去重,提示重复行号 |
| 学号唯一范围 | 单校部署时校内全局唯一即可;若将来多校,需加 `schoolId` 与学号做联合唯一 |

> **A 方案选型说明**:一期在"学号已在别的班"时选择"直接跳过 + 提示"而非"拦截要求确认",原因是简单、不会误删数据,适合"班级名单已基本固定"的场景;转班这类低频操作交由老师手动完成即可。

## 7. 导入模板设计

单 sheet,仅两列:

```
学号 | 姓名
```

模板增强:首行冻结并加批注说明(学号须为 9–10 位、班级由页面决定无需填写);班级不出现在模板中。

校验规则:

- `学号`:非空;长度/格式符合 9–10 位数字;文件内不重复。
- `姓名`:非空。
- 班级:取自页面上下文(导入接口参数),非模板字段。

## 8. 批量删除设计

老师误导入后,需在前端勾选学生 → 批量删除。设计要点如下。

### 8.1 删除语义:删的是"学生身份"

在三表模型中,"删除"有两种语义:**移出班级**(删 `Enrollment`,学生身份仍在)与**删除学生身份**(删 `Student`,人从系统消失)。

误导入的纠正场景,正确语义是**删除学生身份**——一期一个学生只属于一个班,误导入的学生本就不该存在于系统,应连身份带关系一起删除才干净。

### 8.2 级联清理:Enrollment 配 `onDelete: Cascade`

删除 `Student` 时,其 `Enrollment` 必须自动删除,否则会遗留指向不存在学生的孤儿关系记录、违反外键。实现方式是在**子表(Enrollment)指向父表的关系侧**配置 `onDelete: Cascade`(见第 4 节模型)。如此删 Student 时 MySQL 外键级联自动清掉其 Enrollment;将来删 Class 时同理自动清掉该班全部 Enrollment。

### 8.3 两类删除入口

一旦后续出现**答题/刷题记录表**(指向 `studentId`),"删除学生"就必须决定其历史记录的去向。为避免误删真实学生的数据,区分两类入口:

| 入口 | 语义 | 实现 |
|---|---|---|
| 误导入纠正(刚导完、未答题) | 硬删除 Student(级联清 Enrollment) | `deleteMany({ where: { id: { in: [...] } } })` |
| 日常管理删学生(可能已有数据) | 软删除 | 标记 `Student.deletedAt`,查询默认过滤 `deletedAt: null` |

**一期实现**:只做**硬删除**满足"误导入纠正"即可。但**现在就把 `deletedAt` 字段埋进 Student 表**(已在第 4 节模型中预留),理由同一贯的冗余原则——加 nullable 字段零成本,待真实学生开始刷题后要加软删除时无需改表。

**答题记录表的删除策略(未来)**:已在数据模型总表中确定：Attempt.studentId 配 onDelete: Restrict（有记录则禁止硬删），从数据库层逼迫前端走"软删除/二次确认",防止一次误操作清空真实数据。

### 8.4 批量删除实现与交互

```ts
await prisma.student.deleteMany({
  where: { id: { in: selectedIds } }
})
// Enrollment 由外键级联自动清除,无需手动删除
```

前端务必加**二次确认**(显示"将删除 N 名学生"),删除属高危操作。

## 9. 账号与默认密码

学生端一期采用简易登录:**学号 + 默认密码 `wxls12345`**。老师导入学生后,系统自动为每个学生创建默认密码。修改密码功能后续实现。

### 9.1 红线:绝不存明文密码

即便是统一的默认密码、即便是一期,也**绝不在数据库存明文**,而存哈希值。原因:学生表一旦泄露,配合公开的学号即是一批可直接登录的账号;且若一期存明文、二期改存哈希,会出现两套逻辑难以收尾。**从一开始就哈希,后续零负担。**

### 9.2 实现:导入时写入默认密码哈希

导入学生时,为每个学生写入**默认密码的 bcrypt 哈希**(而非明文 `wxls12345`):

```ts
import bcrypt from "bcryptjs"

const DEFAULT_PWD = "wxls12345"
const hash = await bcrypt.hash(DEFAULT_PWD, 10)  // 默认密码相同,整批只需算一次

// createMany 时每个学生都带上 passwordHash: hash
```

登录时用 `bcrypt.compare(输入密码, passwordHash)` 校验。

**性能注意**:默认密码全部相同,因此**整批只需 `bcrypt.hash` 一次**,将结果填给所有学生;切勿逐人各算一次(数百人逐个 bcrypt 会有明显耗时)。

### 9.3 `mustResetPwd` 预留字段

模型中预留 `mustResetPwd: Boolean @default(true)`(见第 4 节)。一期无需使用(恒为 true 亦可),但后续做"修改密码"时,可借它实现"首次登录强制改密"或"提示该生仍用默认密码",属现在埋零成本、后续省改表。

### 9.4 安全声明(一期权宜)

需明确记录:**统一默认密码 + 学号公开,本质上属弱安全方案**——知道学号者即可登录他人账号。一期作为"先跑通"可接受,但应作为**已知一期权宜设计**标注;二期的"修改密码"应配合"首次登录强制改密"一并实现,收掉此安全口子。

## 10. 一期 → 后续(竞赛班/跨班)的改动量

后续放开"学生可属于多个班"时,改动极小:

| 部分 | 一期 | 后续放开多班 | 是否"大改" |
|---|---|---|---|
| 数据库结构 | 三表多对多 | **完全不变** | 否 |
| `@@unique([studentId, classId])` | 已有 | 保留 | 否 |
| `Enrollment.role` | 注释预留 | 取消注释启用 | 否(仅加字段) |
| 导入校验 | 含"已有班级则跳过" | **去掉该校验**(或改为"竞赛班不拦截") | 一处逻辑开关 |
| 导入主流程 | upsert 学生 + 建关系 | 完全复用 | 否 |

即:后续扩展约等于 **取消一行字段注释 + 调整一处导入校验分支**,数据库零迁移、导入管线零重写。

---

## 11. 与题库导入的复用关系

学生导入复用题库导入的同一套 **dry-run 预检管线**:解析 → 内存去重 → 逐行校验(不写库)→ 返回"可导入 N / 跳过 M / 异常 K"预览 → 确认后批量 `createMany(skipDuplicates)` 入库 → 结果回显。批量写入、跳过去重、N+1 规避等性能实践与题库方案一致。
