# 部署文档 —— 智慧题库与智能练测系统

> 基于 Docker Compose 的生产部署：`db`(MySQL 8) + `migrate`(一次性迁移) + `app`(Next.js standalone) + `nginx`(反向代理)。
> 本文档覆盖：架构 → 前置条件 → 首次部署 → 升级 → 备份 → 排障 → 安全加固清单。

---

## 1. 架构总览

```
                    ┌─────────────────────────────────────────┐
   浏览器 ──80/443──▶│ nginx  (反向代理 / gzip / 静态缓存)        │
                    └───────────────┬─────────────────────────┘
                                    │ proxy_pass app:3000
                    ┌───────────────▼─────────────────────────┐
                    │ app    (Next.js standalone, 非 root)      │
                    │        仅 expose 3000，不直接对外          │
                    └───────────────┬─────────────────────────┘
                                    │ DATABASE_URL → db:3306
   migrate(一次性) ─迁移+建管理员──▶ ┌▼─────────────────────────┐
   跑完即退出                       │ db     (MySQL 8, utf8mb4)  │
                                    │        数据存于命名卷 db_data│
                                    └───────────────────────────┘
```

**服务职责**

| 服务 | 镜像/构建 | 作用 | 对外端口 |
|---|---|---|---|
| `db` | `mysql:8.0` | 数据库，数据持久化到命名卷 `db_data` | 无（仅容器内网） |
| `migrate` | 本项目 `migrator` 阶段 | 启动时执行 `prisma migrate deploy` + 初始化管理员，**跑完即退出** | 无 |
| `app` | 本项目 `runner` 阶段 | 运行 Next.js 服务 | 无（仅 `expose 3000` 给 nginx） |
| `nginx` | `nginx:1.27-alpine` | 反向代理、gzip、静态资源缓存、上传体积放宽 | **80**（可选 443） |

**为什么把迁移拆成独立服务？**
Prisma 6 的 CLI 依赖若干被提升到顶层的传递依赖（如 `effect`），无法只靠 standalone 的精简 `node_modules` 运行。因此：
- `app` 镜像保持精简（standalone，不含 Prisma CLI）——**这是真正对外运行、需要控制体积的镜像（≈560MB）**；
- `migrate` 用带完整依赖的镜像执行迁移，跑完退出，`app` 通过 `depends_on: service_completed_successfully` 等它成功后再启动。

## 2. 前置条件

- 一台 Linux 服务器（2C/2G 起步；建议 4G）或本机，安装 **Docker Engine ≥ 24** 与 **Docker Compose v2**。
- 开放入站端口 **80**（启用 HTTPS 再加 **443**）。
- 代码已拉到服务器，进入项目根目录（含 `docker-compose.yml`）。

验证环境：

```bash
docker --version
docker compose version
```

## 3. 首次部署（5 步）

### 3.1 准备环境变量

```bash
cp .env.production.example .env.production
```

编辑 `.env.production`，**务必修改所有口令与密钥**：

```dotenv
# MySQL（db 服务）
MYSQL_ROOT_PASSWORD=<强口令-root>
MYSQL_DATABASE=exam_system
MYSQL_USER=exam
MYSQL_PASSWORD=<强口令-exam>

# 应用（app / migrate 服务）
# ⚠ 主机名必须是 compose 服务名 db；账号/密码/库名要与上面 MYSQL_* 完全一致
DATABASE_URL="mysql://exam:<强口令-exam>@db:3306/exam_system"

# 会话密钥：用下面命令生成后替换
NEXTAUTH_SECRET="<粘贴 openssl 生成结果>"

# 浏览器访问地址（域名或公网/内网 IP，含端口）
NEXTAUTH_URL="http://<你的域名或IP>"
```

生成随机会话密钥：

```bash
openssl rand -base64 32
```

> `.env.production` 已被 `.gitignore` 忽略，**切勿提交到仓库**。

### 3.2 构建并启动

```bash
docker compose up -d --build
```

Compose 会按顺序：`db`(等健康) → `migrate`(迁移+建管理员，退出) → `app`(启动) → `nginx`。

### 3.3 观察启动过程

```bash
docker compose logs -f migrate   # 应看到「所有迁移已应用」+「管理员就绪 admin01/admin02」
docker compose logs -f app       # 应看到「✓ Ready」
docker compose ps                # db=healthy, migrate=exited(0), app/nginx=running
```

### 3.4 访问系统

浏览器打开 `http://<你的域名或IP>/`，或按角色进入：

| 端 | 地址 |
|---|---|
| 教师端登录 | `/teacher/login` |
| 学生端登录 | `/student/login` |
| 超管端登录 | `/admin/login` |

### 3.5 初始账号

数据库初始化后**仅有两个超管账号**：

| 用户名 | 密码 | 权限 |
|---|---|---|
| `admin01` | `wxlysm` | 超级管理员 |
| `admin02` | `wxlysm` | 超级管理员 |

> **首次登录后请立即修改密码**（系统内「修改密码」功能）。此后重启不会覆盖已改密码（初始化脚本对已存在账号是幂等 no-op）。教师、学生账号由超管/教师在系统内创建或导入。

## 4. 常用运维命令

```bash
# 查看状态 / 日志
docker compose ps
docker compose logs -f app

# 停止（保留数据）
docker compose stop

# 启动
docker compose start

# 停止并移除容器（保留数据卷 db_data）
docker compose down

# ⚠ 连同数据库数据一起删除（清库重来时才用）
docker compose down -v
```

## 5. 升级发布（拉取新代码后）

```bash
git pull
docker compose up -d --build      # 重新构建 app/migrate；migrate 自动跑增量迁移
docker image prune -f             # 清理悬空镜像，回收磁盘
```

迁移是增量的：`migrate` 服务每次启动都会 `prisma migrate deploy`，只应用尚未执行的迁移，已应用的跳过。

## 6. 数据备份与恢复

**备份**（导出为 SQL，`*.sql` 已在 `.gitignore` 中）：

```bash
mkdir -p backup
docker compose exec -T db sh -c \
  'mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" --default-character-set=utf8mb4 exam_system' \
  > backup/exam_system_$(date +%Y%m%d_%H%M%S).sql
```

**恢复**：

```bash
docker compose exec -T db sh -c \
  'mysql -uroot -p"$MYSQL_ROOT_PASSWORD" --default-character-set=utf8mb4 exam_system' \
  < backup/exam_system_20260701_120000.sql
```

> 数据实际存放在命名卷 `wxlysm-exam-sys_db_data`，只要不执行 `down -v` 就不会丢。建议将 `backup/` 目录定期同步到异地或对象存储，并用 cron 定时执行上面的备份命令。

## 7. 启用 HTTPS（可选，生产强烈建议）

1. 准备证书（Let's Encrypt / 企业证书），放到 `nginx/certs/`（该目录已被 `.gitignore` 忽略）：
   - `nginx/certs/fullchain.pem`
   - `nginx/certs/privkey.pem`
2. 在 `docker-compose.yml` 的 `nginx` 服务放开：
   - `ports:` 增加 `- "443:443"`
   - `volumes:` 增加 `- ./nginx/certs:/etc/nginx/certs:ro`
3. 编辑 `nginx/conf.d/default.conf`，取消文件末尾 HTTPS server 块与 80→443 跳转块的注释，把 `server_name` 改成你的域名。
4. 把 `.env.production` 的 `NEXTAUTH_URL` 改成 `https://你的域名`。
5. 重启：`docker compose up -d`。

## 8. 排障速查

| 现象 | 排查 |
|---|---|
| `app` 起不来、日志报连不上库 | 确认 `DATABASE_URL` 主机名是 `db`、账号密码与 `MYSQL_*` 一致；`docker compose logs db` 看是否 healthy |
| nginx 报 `Bind for 0.0.0.0:80 failed: port is already allocated` | 宿主机 80 端口被占用（另一个服务/容器）。释放占用，或改 `docker-compose.yml` 里 nginx 端口为 `"8080:80"` |
| 登录后立刻被登出 / 回调异常 | `NEXTAUTH_URL` 必须与浏览器实际访问地址一致；经代理时 nginx 要透传 `X-Forwarded-Proto`（默认配置已带） |
| `migrate` 退出码非 0 | `docker compose logs migrate` 看具体报错；多为库连接或权限问题 |
| 中文显示乱码 | 数据库以 utf8mb4 建库（compose 已配 `--character-set-server=utf8mb4`）；用 mysql 客户端查库时加 `--default-character-set=utf8mb4` |
| Excel 导入大文件 413 | nginx 已设 `client_max_body_size 20m`，更大文件按需调高 |

查看迁移是否全部应用：

```bash
docker compose run --rm migrate node node_modules/prisma/build/index.js migrate status
```

## 9. 体积与最佳实践说明

- **多阶段构建**：`deps → builder/migrator → runner`，最终对外的 `app` 镜像只含 standalone 运行产物（≈560MB，基于 `node:22-slim`）。
- **`.dockerignore`**：`node_modules`、`.next`、`.git`、`document/`、`design/`、`.env*` 等都不进 build context，构建更快、镜像更干净。
- **standalone 输出**：`next.config.ts` 已配 `output: "standalone"`，自动裁剪出运行所需的最小 `node_modules` 子集。
- **非 root 运行**：`app` 以 `nextjs(uid1001)` 用户运行，降低容器逃逸风险。
- **Debian slim 而非 alpine**：规避 Prisma 在 musl/openssl 上的引擎兼容问题，换取零折腾（体积略大但稳定）。
- **数据库不对外暴露**：`db` 不映射宿主端口，仅容器内网可达；`app` 也只 `expose` 给 nginx。
- **迁移与运行分离**：见 §1，`app` 镜像因此免带 Prisma CLI，更小更安全。

> `migrate` 镜像约 1.4GB，但它**只在本机构建、跑完即退**，不需要推送到镜像仓库，不影响线上运行体积。
